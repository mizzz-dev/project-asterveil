# RELEASE SCOPE OPTIONS

本書は、現状の公開実装を起点に、どのリリース形態を先に狙うべきかを比較する。

## 1. スコープ比較（最小構成）

### 1.1 Steam デモ
- 必要最小構成
  - 1〜2時間の完結体験（導入、戦闘、クエスト、ボス）
  - キーボード/パッド操作
  - クラッシュ時の再開可能なセーブ
  - ストア素材（説明文、画像、トレーラー最小）
- メリット
  - 開発リスクを抑えつつ公開検証できる
  - ウィッシュリスト獲得を先行できる
- デメリット
  - 売上化は限定的
  - 体験範囲が狭く、評価がコンテンツ量に引っ張られる

### 1.2 Steam Early Access
- 必要最小構成
  - 章単位で継続プレイ可能なボリューム
  - 更新ロードマップ公開
  - セーブ互換ポリシーと既知課題の明記
  - クラッシュ収集とサポート窓口
- メリット
  - 収益化とユーザーフィードバックを並行できる
  - 長編開発と相性が良い
- デメリット
  - 継続運用コストが高い
  - 品質事故時のレビュー影響が長期化しやすい

### 1.3 フル1.0
- 必要最小構成
  - 本編完結相当のコンテンツ量
  - 難易度調整、チュートリアル、設定、ローカライズ
  - QA回帰体制、運用監視、パッチ運用
- メリット
  - 完成品としての訴求がしやすい
- デメリット
  - 初回リリースまでの期間・資金リスクが最大

### 1.4 App Store / Google Play 同時展開
- 追加要件
  - タッチUI最適化、解像度/端末差異対応
  - ストア審査要件（権限、課金、プライバシー）
  - 端末性能別最適化とクラッシュ率管理
- メリット
  - 市場規模が拡大
- デメリット
  - 審査・検証コストが急増し、Steam先行の速度が落ちる

## 2. 論点別の推奨判断

- **Pythonベース現行実装 → 提出可能クライアント**
  - まずSteam向けに単一配布経路を確立し、提出手順を固定してからモバイル拡張する。
- **Steam先行 vs モバイル同時**
  - 初回はSteam先行を推奨。同時展開はQA負荷が高く、現状体制では遅延リスクが大きい。
- **フレンド協力機能**
  - 初回リリース必須から外し、Early Access以降の段階投入（限定モード）を推奨。
- **長編シナリオ/エンドコンテンツ**
  - 初回は縦切り完結体験を優先し、全量実装は1.0側へ計画的に後ろ倒し。

## 3. 現状から最も現実的な推奨スコープ

- **推奨: Steam デモ → Steam Early Access の2段階リリース**
  1. Steamデモで操作性・戦闘理解・クラッシュ率を検証
  2. セーブ互換/運用監視を固めてEarly Accessへ移行
- 理由
  - 現状は Internal Playable であり、1.0直行より段階公開の方が失敗コストを抑えられる。
  - PR #54までの機能拡張は「遊びの厚み」寄りで、配布運用面の整備はこれからのため。
