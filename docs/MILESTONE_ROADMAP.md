# Milestone Roadmap (0〜6 months)

本書は、MVP達成までの月次到達点とGo/No-Go判断を管理するためのロードマップである。

## 1) フェーズ区分

- **Prototype（M0-M2）**: 戦闘/操作/章導入の成立確認
- **Pre-Vertical Slice（M3-M4）**: 1章縦切り + 開発運用の再現性確立
- **Vertical Slice（M5-M6）**: 3章相当MVP + Go/No-Go評価

## 2) 月次マイルストーン

| 月 | フェーズ | 主要到達点 | 完成しているべき成果物 |
|---|---|---|---|
| M0 | Prototype | キックオフ、開発環境・規約確定 | 命名規約、ブランチ運用、KPI定義書 |
| M1 | Prototype | コア戦闘試作（弱点/ブレイク） | 戦闘プロト、簡易HUD、初期テスト項目 |
| M2 | Prototype | UI/操作試作 + 章1設計凍結 | 章1仕様、チュートリアル導線、操作計測ログ |
| M3 | Pre-VS | 1章縦切り実装（探索→ボス） | Playable Chapter 1、進行フラグ管理、QA観点表 |
| M4 | Pre-VS | 量産性検証（章2着手、クエスト量産） | 章テンプレ、クエストテンプレ、データ投入手順 |
| M5 | Vertical Slice | 3章相当の品質統合 | 章1-3プレイ可能版、育成最小ループ、KPIダッシュボード |
| M6 | Vertical Slice | Go/No-Go判定 | 判定レポート、残課題一覧、次フェーズ計画案 |

## 3) フェーズ目標

### Prototype（M0-M2）
- 戦闘の「理解できる手応え」を成立させる。
- モバイルUI成立可能性（操作回数/視認性）を測る。
- 章1仕様を凍結し、実装へ引き渡す。

### Pre-Vertical Slice（M3-M4）
- 1章縦切りで進行不能ゼロを達成。
- 章/クエストの量産テンプレを確立。
- 実装→QA→修正の反復時間を測定。

### Vertical Slice（M5-M6）
- 3章分でMVP体験を提供。
- KPIをもとに6か月時点のGo/No-Goを判断。
- Full Productionへ進む場合の増員/外注計画を確定。

## 4) Go / No-Go判断基準（再整理）

### Go条件（全て満たす）
- 章1クリア率70%以上
- 戦闘理解アンケート4.0/5以上
- 重大バグ（進行不能/セーブ破損）クローズ済み
- 章追加の見積誤差±20%以内

### No-Go / 保留条件（いずれか該当）
- 体験KPIが連続2スプリント未達
- コア戦闘がUI依存で理解不能（改善施策後も未解消）
- セーブ互換性が安定せず、品質リスクが高い

### 判定後アクション
- **Go**: Full Productionへ移行し、Should項目を段階追加。
- **保留**: Must項目に集中して2か月再検証。
- **No-Go**: スコープ再定義（章数縮小 or 協力機能延期）を経営判断に上申。

## 5) 遅延しやすいポイント

- シナリオ演出要求の肥大化
- UI改修の再発（情報過多、入力差異）
- データ投入手作業によるボトルネック
- マルチ同期不具合のデバッグ長期化

## 6) スコープ調整の判断基準

1. プレイヤー価値より実装都合が優先されていないかを確認。
2. Mustの品質要件（進行可能・理解可能・保存可能）を最優先。
3. Should/Couldは「KPI改善寄与」が示せない場合は延期。
4. 高コスト演出はVertical Slice以降へ後ろ倒し。

## 7) issue / sprint planning への接続

- 各月の成果物をEpicの完了条件へ紐付ける。
- スプリント計画では「機能」ではなく「検証仮説」を単位化する。
- 週次でKPIレビューを実施し、翌スプリントの優先順位を更新する。
