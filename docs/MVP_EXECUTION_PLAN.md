# MVP Execution Plan (Project Asterveil)

本書は `DESIGN_PROPOSAL_JRPG.md` を実装可能な初期計画に落とし込み、6か月以内のGo/No-Go判断へ直結させるための実行計画である。

## 1) MVPの目的

- **目的A: コア体験の成立検証**
  - 「続きが気になる物語」と「深いが理解できるコマンド戦闘」が同時成立するかを検証する。
- **目的B: 本編開発の量産可能性検証**
  - 章実装・クエスト実装・バトル調整・QAを反復できるパイプラインを確立する。
- **目的C: プラットフォーム現実性検証**
  - Steam優先開発を前提に、モバイルUI/性能課題を早期に可視化する。

## 2) MVPに含める機能（In Scope）

### 2.1 コンテンツ範囲
- メインストーリー **3章分**（章1〜3）
- プレイアブル仲間 **4名**（主人公含む）
- ボス戦 **3体**（章ごと1体）
- サブクエスト **20件（テンプレ再利用前提）**

### 2.2 バトル範囲
- CTB行動順（SPD + Weight）
- 弱点（属性 + 武器種）
- ブレイク / スタン / 連携の最小成立
- 状態異常6種（提案書準拠）
- 難易度3段階（Story / Standard / Veteran）

### 2.3 システム範囲
- 育成最小ループ（Lv・星紋ボード簡易版・装備）
- セーブ/ロード（オート3 + 手動10の土台）
- テレメトリ最小導入（章クリア率、全滅率、平均戦闘ターン）
- 協力バトル試験版1種（2〜4人、限定ルール）

### 2.4 UX範囲
- PC（キーボード/パッド）基本操作
- スマホ想定UIの主要画面検証（戦闘・編成・クエスト導線）
- チュートリアルの分割導入（戦闘基礎→応用）

## 3) MVPから除外する機能（Out of Scope）

- 12章本編のフル実装
- 仲間8〜10名体制
- エンドコンテンツ12種の本実装
- ギルド拡張、シーズン運営、本格LiveOps
- クロスセーブ本番運用（設計のみ）
- 高コスト演出（フルボイス大規模化、シネマティック量産）

## 4) Vertical Sliceで先に検証する内容

Vertical Sliceは「品質付きで再現可能な1章分の完成サンプル」と定義する。

- 探索→会話→通常戦→ボス→章クリアまでの一連体験
- 主要UI（戦闘HUD、行動順、弱点表示、予兆表示）
- 章中クエスト導線（メイン + サブ + 個別）
- クリア後リザルトと次章フック演出
- QA受け入れ基準に沿ったテスト実施（進行不能・セーブ破損・UI崩れ）

## 5) なぜこの範囲が妥当か

- 提案書で定義された「最初の6か月」および「MVP（3章/仲間4名/サブ20/協力試験）」に一致する。
- 12章を目指す前に、1章の再現性（量産可能性）を証明する方が全体リスクを抑えられる。
- マルチは試験版に限定することで、本編品質（戦闘・物語・UI）を優先できる。

## 6) 想定プレイフロー（MVP版）

1. チュートリアル戦闘（弱点/ブレイク基礎）
2. 章1導入（故郷消失イベント）
3. フィールド探索 + サブクエ受注
4. 通常戦闘で育成素材獲得
5. 章ボス（予兆対応 + ブレイク活用）
6. 章クリア → 仲間強化開放
7. 章2以降で行動順制御・連携を拡張学習
8. 章3完了後、協力試験コンテンツ解放

## 7) 成功条件（MVP Exit Criteria）

### 7.1 体験KPI
- 章1クリア率: **70%以上**
- ボス再挑戦継続率: **50%以上**（挑戦後即離脱しない）
- 戦闘理解アンケート: **4.0/5以上**
- UI操作ストレス低評価: **20%未満**

### 7.2 開発KPI
- 1章相当の実装サイクル（仕様→実装→QA→修正）を**6週間以内**で再現
- 重大バグ（進行不能/セーブ破損）を各マイルストーンで**0件クローズ状態**
- テレメトリ欠損率 **5%未満**

## 8) 技術的検証項目

- バトルDSL/データ駆動化でスキル追加工数が増えないか
- セーブ世代管理 + CRCの復旧導線が実運用可能か
- マルチ試験で同期破綻が許容範囲か（再接続復帰含む）
- モバイル中位端末で発熱/フレーム安定性を維持できるか
- Addressables想定で章追加時のパッチサイズを制御できるか

## 9) ゲームデザイン上の検証項目

- ブレイク・弱点・連携が「理解できるが奥深い」難度に収まるか
- 章終盤の感情ピーク（喪失/継承）をプレイヤーが認知するか
- 仲間ロール差（盾/支援/火力）が編成選択に効いているか
- サブクエが単純なお使い化を回避できているか

## 10) UI/UX上の検証項目

- 行動順タイムラインの視認性（PC/モバイル）
- 2階層以内コマンドの原則遵守
- 予兆UIで失敗理由が理解できるか
- オート/倍速利用時も学習機会が失われないか

## 11) QA観点で先に潰すべき項目

- 進行不能（イベントフラグ競合、クエスト依存欠落）
- セーブ互換/破損（バージョン差分、中断復帰）
- UI崩れ（解像度、長文ローカライズ、入力切替）
- バトル不整合（弱点判定、行動順逆転、状態異常永続化）
- 通信切断復帰（協力試験版）

## 12) 6か月Go/No-Go直結チェックポイント

| 月 | チェックポイント | 閾値 | 判定失敗時のアクション |
|---|---|---|---|
| M2 | コア戦闘理解 | チュートリアル後の勝率60%以上 | 戦闘要素を削減し、弱点/ブレイク説明を再設計 |
| M3 | 1章縦切り成立 | 章クリアまで進行不能0件 | イベント構造を再分割し、演出を簡素化 |
| M4 | 量産性確認 | 章追加の見積誤差±20%以内 | 章テンプレート化を優先、演出粒度を統一 |
| M5 | Vertical Slice品質 | QA重大バグ密度目標達成 | Should/Couldを凍結しMustへ再集中 |
| M6 | MVP総合評価 | 主要KPIを4/5以上達成 | Full Production移行を保留し再設計 |

## 13) issue / sprint ticket化のためのテンプレート

各機能は以下の最小単位でチケット化する。

- **User Story**: 誰が/何を/なぜ
- **Acceptance Criteria**: Given/When/Then 3〜5項目
- **Dependencies**: 前提データ、前工程タスク
- **Test Cases**: 正常/異常/境界
- **Telemetry**: 計測イベント名と成功条件
